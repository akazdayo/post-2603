# このポスターに足りないもの

`main.typ` だけを見ると、いちばん足りないのは「どこまで成功したのかを、短時間で納得させる情報」です。背景と目的は十分ありますが、評価と結論の見せ方がまだ弱いです。

- 結論の一文。今は [主張](./main.typ) が研究の狙いで、読者が最初に知りたい「結局できたのか」は [結果](./main.typ) まで行かないと分かりません。タイトル直下に「Webカメラのみで腰・脚の位置追従は可能、ただし回転情報と遅延が課題」くらいの要約が欲しいです。
- 実験条件。 [手法](./main.typ) と [結果](./main.typ) には、カメラ性能、PC/GPU、FPS、被験者数、撮影距離、照明条件、VRChat 実行条件がありません。`51.17 ms` と `85.18 ms` の数字の重みが伝わりにくいです。
- 定量評価。 [結果](./main.typ) の「比較的良好」「一定程度再現」は曖昧です。位置誤差、追従遅延、成功率、破綻しやすい動作など、1つでも数値指標があると説得力が上がります。
- 比較表。 [比較](./main.typ) は文章だけなので、物理トラッカー方式との違いが流し読みで拾えません。`コスト / 導入手間 / 位置精度 / 回転取得 / 遮蔽耐性 / 遅延` の表があると強いです。
- 手法の図解。 [手法](./main.typ) にデモ画像はありますが、`Webカメラ → MeTRAbs → 座標補正 → Godot → VRChat OSC` の処理フロー図がありません。ここは文章より図のほうが伝わります。
- 失敗例。 [結果](./main.typ) で回転情報不足や遅延を述べていますが、破綻した姿勢や遮蔽時の例がないです。成功例だけでなく失敗例を1枚入れると誠実で分かりやすいです。
- 参考文献の厚み。 [参考文献](./main.typ) がほぼ GitHub URL だけです。MeTRAbs の論文や、比較対象の代表的な FBT 手法を 1-2 件入れたほうが研究ポスターとして締まります。
- 発表者情報かデモ導線。今は GitHub QR のみです。もし用意できるなら、リポジトリより「動作動画」の QR のほうが見に来た人には効きます。

優先順位をつけるなら、まず `結論の一文`、`実験条件`、`比較表` の3つです。ここを足すだけで、かなり研究ポスターらしくなります。
